# AI Agent Project Rules

> ใช้ไฟล์นี้เป็นกติกากลางสำหรับทุกโปรเจค
> ห้ามใส่ข้อมูลเฉพาะโปรเจคลงในไฟล์นี้
> ข้อมูลเฉพาะโปรเจคให้ใส่ใน `project.md`, `README.md`, หรือเอกสารเฉพาะทางเท่านั้น

## 1. ภาษาและสไตล์การตอบ

- ตอบเป็นภาษาไทยเป็นหลัก ยกเว้นผู้ใช้ขอภาษาอื่น
- ตอบสั้น กระชับ แต่ต้องไม่ตัดข้อมูลสำคัญ
- ถ้าไม่แน่ใจ ให้ถามก่อน อย่าเดาแล้วสร้างไฟล์หรือโค้ดจำนวนมาก
- ถ้าต้องสรุป ให้แยกเป็นหัวข้อสั้น ๆ อ่านง่าย
- หลีกเลี่ยงการสร้างแผนยาวเกินจำเป็น เว้นแต่ผู้ใช้ขอ

## 2. กฎก่อนสร้างหรือแก้ไฟล์

- อ่านไฟล์เดิมก่อนเสมอ
- ห้ามเขียนทับไฟล์เดิมโดยไม่จำเป็น
- ถ้าจะเขียนทับไฟล์สำคัญ ต้องถามผู้ใช้ก่อน
- ถ้าข้อมูลไม่ครบ ให้ถามผู้ใช้ก่อนสร้างไฟล์หลัก
- ถ้าไม่มีคำถาม ให้บอกว่าไม่มีคำถาม แล้วสรุปไฟล์ที่จะสร้างก่อนลงมือ

## 3. ไฟล์หลักของโปรเจค

- `rule.md` = กฎกลางของ AI agent ในโปรเจคนี้
- `project.md` = รายละเอียดเชิงลึกของโปรเจค ใช้เป็นแหล่งความจริงหลัก
- `README.md` = หน้า GitHub อ่านง่ายสำหรับคนทั่วไป
- `hardware_lib.md` = รายละเอียด hardware และ library
- `platformio.ini` = config ของ PlatformIO ถ้าใช้ PlatformIO
- `src/main.cpp` = entry point ของ firmware
- `.gitignore` = ignore build output และไฟล์ชั่วคราว
- `.agent/workflows/` = note คำสั่ง upload/monitor ถ้าผู้ใช้ต้องการ

## 4. กฎของ `project.md`

- ต้องบอกเป้าหมาย hardware pin map library และสถานะล่าสุด
- ถ้ายังไม่รู้ข้อมูล ให้ใส่ `TBD` หรือ `Open Question`
- ถ้ามีการเปลี่ยน pin, hardware, library, config หรือ behavior สำคัญ ต้องอัปเดตไฟล์นี้
- ห้ามใช้ `project.md` เป็นหน้าโชว์ GitHub ให้ใช้ `README.md` แทน

## 5. กฎของ `README.md`

- ต้องอ่านง่าย เหมาะกับ GitHub
- อธิบายว่าโปรเจคคืออะไร ใช้ hardware อะไร และเริ่มใช้งานอย่างไร
- ไม่ต้องละเอียดเท่า `project.md`
- ห้ามใส่ secret, token, password หรือ private note
- ถ้ายังเป็น prototype ให้บอกสถานะตรง ๆ

## 6. กฎด้านโค้ด

- เริ่มจากโครงสร้างเรียบง่ายก่อน
- แยก module เมื่อ logic เริ่มใหญ่หรือเริ่มซ้ำ
- ใช้ชื่อไฟล์ ตัวแปร และฟังก์ชันแบบ `snake_case`
- ใช้ค่าคงที่แบบ `UPPER_SNAKE_CASE`
- ใช้ `#pragma once` สำหรับ header
- หลีกเลี่ยง `delay()` ใน logic หลัก ให้ใช้ `millis()` หรือ timer เมื่อเหมาะสม
- `main.cpp` จะต้องมีโค้ดสั้นที่สุด เน้นแค่การเรียกใช้ฟังก์ชันจาก module อื่น ถ้ามีลอจิกข้าม module ค่อยนำมาเขียนประสานใน main

## 7. กฎด้าน Embedded / Hardware

- ระวัง boot pin, strapping pin, pin ที่มี LED หรือวงจรบนบอร์ดพ่วง
- ระบุ pin direction ให้ชัด เช่น input, input pull-up, output, open-drain
- ห้ามให้ GPIO ที่คุม MOSFET, relay, power path หรือ load สำคัญกระตุกตอน boot
- ถ้ามีวงจรวัดไฟหรือกระแส ต้องระบุข้อจำกัดด้านแรงดัน กระแส fuse และ safety
- ถ้าเกี่ยวกับไฟบ้าน AC หรือแรงดันอันตราย ต้องเตือนชัดเจนและไม่เสนอวงจรต่อตรง

## 8. กฎด้านคำสั่ง Terminal

- ห้ามรัน upload/flash/monitor แทนผู้ใช้ ถ้าผู้ใช้ไม่ได้อนุญาตชัดเจน
- ถ้าต้องให้ผู้ใช้รันคำสั่ง ให้เขียนเป็น command block สั้น ๆ
- ห้ามใช้คำสั่งลบหรือเขียนทับไฟล์สำคัญโดยไม่ยืนยัน

## 9. เมื่อมีคำถาม ให้ถามแบบนี้

```text
ก่อนสร้างไฟล์ ผมขอเคลียร์ 3 เรื่อง:
1. ...
2. ...
3. ...
```

ถ้าไม่มีคำถาม:

```text
ไม่มีคำถามเพิ่มครับ ผมจะสร้าง/แก้ไฟล์ต่อไปนี้: ...
ยืนยันให้ทำได้เลยไหม?
```

## 10. สิ่งที่ห้ามทำ

- ห้ามสร้าง `project.md` แบบเดาเองทั้งไฟล์
- ห้ามสร้างไฟล์ template จำนวนมากถ้าผู้ใช้ยังไม่ขอหรือยังไม่ยืนยัน
- ห้ามเขียนทับไฟล์เดิมโดยไม่อ่านก่อน
- ห้ามเปลี่ยน pin map เองโดยไม่บันทึกเหตุผล
- ห้ามใส่ข้อมูล private ลง README
- ห้ามทำให้ `rule.md` กลายเป็นเอกสารเฉพาะโปรเจค
